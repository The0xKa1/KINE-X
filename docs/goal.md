# Goal

## 一句话目标

KINE//X 要把短视频动作教学变成可交互的实时动作外挂。
用户输入标准动作视频。
系统输出可旋转、可慢放、可比对的虚拟动作教具。
评委打开网页或扫码即可跟练。
前端实时展示动作匹配度、关节误差和安全风险。

## 核心痛点

用户收藏了大量动作教学视频，但缺少练习反馈。
单一视频视角无法解释复杂动作。
错误动作可能导致受伤。
普通用户看不到骨骼角度和重心偏差。
真实 3D HMR 模型现场运行成本高。
前后端如果耦合过深，黑客松合体联调容易崩。

## 最终用户体验

页面首屏必须立即出现动作舞台。
系统初始化时展示优雅 loading。
默认展示半透明科技感动作教练。
用户可以选择动作种子。
用户可以切换 front、side、top 视角。
用户可以控制播放、暂停、速度和进度。
用户可以打开摄像头进入跟练。
摄像头画面必须符合镜子直觉。
3D 教练画布不镜像，保持面对面呈现。
页面实时展示综合匹配分数。
页面实时展示 Combo 反馈。
页面实时展示关节角度误差。
页面实时展示 3D 欧氏距离误差。
页面突出高风险关节和动作偏差。

## 黑客松演示目标

评委无需安装客户端。
网页在手机和笔记本上都能打开。
前端即使没有真实后端也能完整演示。
后端一旦提供 WebSocket 帧，前端能快速切换到真实数据。
演示故事必须清楚：短视频输入、动作 DNA、实时跟练、风险反馈。
大屏展示时能看出 AI、3D、实时互动和游戏化反馈。

## 产品目标

把短视频从收藏内容变成可执行练习。
把动作教学从观看模仿升级为实时反馈训练。
把 SMPL / HMR 模型结果包装成普通用户可理解的界面。
把专业风险提示变成即时可见的微反馈。
让健身、传统运动、舞蹈和球类动作共用统一动作 DNA。
为后续训练计划、内容平台和付费课程留下空间。

## 技术目标

前端和后端彻底解耦。
前端使用 TypeScript 维护数据契约。
重模型在后端运行。
前端只消费轻量动作帧。
高频帧数据进入 FrameBuffer。
UI 组件不直接管理高频骨骼帧。
渲染由 RAF 主循环主动拉取最新帧。
旋转统一使用 Quaternion。
空间坐标统一使用米。
坐标系统一为右手系。
摄像头流和 3D 舞台并列管理。
UI 组件和渲染核心并列管理。
切换动作时显式释放旧资源。

## 后端对齐目标

FastAPI 负责动作 DNA 管理和 WebSocket 分发。
模型管线负责提取标准视频的 seedJoints。
现场摄像头可先由前端 MediaPipe 计算 evaluator joints。
后端按 `FRAME_STREAM` 协议推送帧数据。
每帧包含时间戳、动作种子、全局位移、局部 Quaternion、关节坐标和评分指标。
后端不需要向前端传完整视频流。
前端不依赖后端模型初始化速度。

## 数据契约目标

`MotionFrame` 是前后端对齐核心。
`seedJoints` 表示标准动作骨架。
`joints` 表示评委或用户动作骨架。
`localRotations` 表示骨骼局部 Quaternion。
`metrics` 表示关节误差和风险。
`score` 表示综合匹配度。
`combo` 表示游戏化连续反馈。
所有坐标必须是米。
所有旋转必须是 `[x, y, z, w]`。

## 验收标准

页面能稳定启动。
首屏有动作舞台和初始化反馈。
动作种子可以切换。
播放、速度和时间轴可以控制。
标准骨架和评委骨架差异可见。
评分、Combo、风险和关节指标会随帧变化。
桌面和手机宽度无横向溢出。
摄像头失败时能回退到 mock stream。
`npm run check` 必须通过。

## 最终呈现

观众看到的是轻量、炫酷、可交互的运动 AI 教练。
评委理解的是一个可落地的前后端分离系统。
团队拥有的是一个能继续接真实模型的前端骨架。
答辩时项目能同时讲清技术深度、产品价值和现场可玩性。
